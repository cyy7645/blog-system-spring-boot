OAuth 2.0 Principles and practices of certification

The benefits of using OAuth 2.0 authentication are clearly visible. You only need to use the same account password to access it on various websites, eliminating the cumbersome process of registering on each website.                              This article introduces the principles of OAuth 2.0 and demonstrates the process of OAuth 2.0 authentication based on Spring Security and GitHub accounts.

What's OAuth 2.0

The specification of OAuth 2.0 can refer to: RFC 6749

OAuth is an open standard that allows users to let third-party applications access private resources (such as photos, videos, contact lists) that the user stores on a website without having to provide a username and password to a third-party application. Currently, the latest version of OAuth is 2.0

OAuth Allow users to provide a token instead of a username and password to access the data they store in a particular service provider. Each token authorizes a particular website (eg, a video editing website) to access a particular resource (eg, just a video in a certain album) for a specific period of time (eg, within the next 2 hours). This way, OAuth Allow users to authorize third-party websites to access information they store on other service providers without sharing their access permissions or all of their data.

The core concept of OAuth 2.0

OAuth 2.0 has four main types of roles:

  • resource owner: resource owner, the "user" of the terminal
  • resource server: The resource server, the service provider, stores protected resources. To access these resources, you need to get an access token (access                                  Token). It can be the same server as the authentication server, or it can be a different server. If we visit the Sina blog site, then if you use the Sina blog account to log in to the Sina blog site, then the Sina blog resource and the Sina blog authentication are the same, can be considered the same server. If we are a Sina blog account to log in to know, then obviously the resources and Sina authentication are not a server.
  • client: Client, a third-party application that represents a resource request to a protected resource.
  • authorization server: Authorization server, after verifying the resource owner and obtaining authorization, the access token will be issued to the client.

OAuth 2.0 certification process

The certification process is as follows：

  • （A) After the user opens the client, the client requests authorization from the resource owner (user).
  • （B) The user agrees to give the client authorization.
  • （C）The client uses the authorization obtained in the previous step to apply for an access token from the authentication server.
  • （D）After the authentication server authenticates the client, it confirms that it is correct and agrees to issue an access token.
  • （E）The client uses the access token to request the resource server to obtain the resource.
  • （F）The resource server confirms that the token is correct and agrees to open the resource to the client.

Among them, user authorization has four modes:

  • authorization code
  • implicit
  • resource owner password credentials
  • client credentials
